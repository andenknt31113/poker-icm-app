# Poker ICM/BF Calculator 仕様書

> 2026-05-08 初版（ユーザー提示の仕様書をそのまま保存）

## 1. 背景・課題

### 課題

- ポーカーのトーナメント（特に店舗トナメ）では、ICM・BF・RPを理解しないと「降り得」「コール得」を判断できないが、実戦中に手計算は不可能。
- 既存の海外ツール（ICMIZER, GTO Wizard, Tournament Cruncher等）は **英語UI / PC前提 / 月額高額** で、日本のレク〜中級プレイヤーには敷居が高い。
- 日本語の専用アプリは現状ほぼゼロ（勝率計算アプリはあるが、ICM/BF対応は未開拓）。

### 解きたいこと

1. **店舗トナメ中に1ハンドだけパッと確認できる**スピード感
2. **日本語で完結**するUI／用語解説
3. **店舗トナメの典型構造をプリセット化**（人数・buy-in・ペイ%）して入力負担を最小化

## 2. ターゲットユーザー

- **メイン**：店舗トナメに月数回〜週1で参加する日本人プレイヤー（レク〜中級）
- **サブ**：オンラインMTTもプレイし、復習用ツールが欲しい中〜上級者
- **想定スキル感**：cEV・ポットオッズは知っている／ICM・BFは「名前は聞いたことある」レベル

## 3. プロダクト原則

1. **3タップで結論まで出す**（入力フォームの認知負荷を最小化）
2. **数字だけでなく "判定" まで出す**（コール / フォールド / 微妙）
3. **学習機能を内蔵**（「なぜこの数字になるのか」を即座に展開できる）
4. **オフライン動作**（店舗のWi-Fi環境を前提にしない）

## 6. 技術仕様

### 6.1 ICM計算ロジック

**Malmuth-Harville アルゴリズム**

```
P(Xが1位) = Xのスタック ÷ 全チップ

P(Xが2位) = Σ_{Y≠X} [ P(Yが1位) × Xのスタック ÷ (全チップ - Yのスタック) ]

P(Xが3位以降) も同様に再帰的に計算

エクイティ = Σ P(X位) × 該当賞金
```

### 6.2 BF計算

```
BF = (現在エクイティ - 負け時エクイティ) ÷ (勝ち時エクイティ - 現在エクイティ)
```

### 6.3 必要勝率計算

```
cEV必要勝率 = call ÷ (call + 勝った時に取れる額)
$EV必要勝率 = (call × BF) ÷ (call × BF + 勝った時に取れる額)
RP = $EV必要勝率 - cEV必要勝率
```

### 6.4 データ構造

| 項目 | 型 | 説明 |
| --- | --- | --- |
| players | Array<{ name, stack }> | 残プレイヤーのスタック配列 |
| payouts | Array<number> | 各順位の賞金（金額または%） |
| heroIndex | number | 自分のplayers配列内のindex |
| villainIndex | number | 勝負相手のindex |
| callAmount | number | コール額（BB） |
| blinds | number | ポットの中の死に金（BB） |

## 7. 技術スタック候補

### 7.1 推奨：React Native + Expo
- iOS / Android 両対応 / TypeScriptで計算ロジック / EAS Build / オフライン動作

### 7.2 別案：Flutter
- パフォーマンス良 / Dartの学習コストあり

### 7.3 さらに別案：Webアプリ（PWA）

## 12. リスク・懸念

- **市場規模が読めない**：日本のポーカー人口は伸びているが、有料アプリにお金を払う層は限定的
- **ストア審査**：ポーカー関連は審査で弾かれることがある（特にiOS）→ 「学習・計算ツール」として申請する戦略
- **ICM計算の精度**：Malmuth-Harvilleはあくまで近似。スキル差・ポジション差は反映されない（既存ツールも同じだが）
- **競合参入**：もしGTO Wizardが日本語化したら一気にシェアを取られる
